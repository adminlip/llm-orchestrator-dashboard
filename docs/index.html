踩坑与解决方案

项目落地过程中遇到的关键问题，以及最终采用的处理方式

问题 01

主模型故障后整体不可用

早期版本依赖单一主模型，一旦请求失败或平台抖动，整条链路中断。

解决方案

实现主备切换机制：记录故障、提升备份模型为主、恢复后可降级回子代理。

问题 02

不同平台接入方式不统一

Hermes / OpenClue / CodeClue 与 OpenAI 兼容平台调用形态存在差异。

解决方案

抽象平台适配层，统一执行上下文与结果结构，支持 Mock 与真实 API 自动切换。

问题 03

演示环境容易暴露真实地址与密钥

直接展示生产地址和真实 API 配置存在安全风险，不适合外部分享。

解决方案

将展示页与真实服务隔离：对外只提供静态介绍页，所有敏感地址替换为示例域名。

问题 04

部署后接口返回异常

云服务器存在端口冲突与运行路径差异，导致健康检查路径异常。

解决方案

改为 systemd 固化启动命令、指定 app-dir 并切换可用端口，最终完成稳定运行。

交付结果
  • 支持主模型动态切换与自动故障切换
  • 支持多平台统一适配与可观测展示面板
  • 支持 Windows / Linux / Docker / systemd 的开箱即用部署
  • 支持对外安全展示（不暴露真实 API 和服务器地址）
